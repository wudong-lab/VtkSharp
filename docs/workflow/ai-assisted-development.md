# VtkSharp AI 辅助开发协作规范

本文档用于约束和复用 `VtkSharp` 项目的 AI 辅助开发流程。目标不是让 AI 替代工程判断，而是让 AI 稳定承担代码生成、局部实现、示例补充、测试辅助和文档整理工作，同时把架构、API 边界、生命周期和质量验收保留在人类开发者控制之下。

## 1. 协作定位

`VtkSharp` 是围绕 VTK 的 .NET/C# 封装项目。AI 协作时应默认把它视为工程图形、三维可视化、桌面软件基础设施项目，而不是普通演示项目。

AI 的主要职责：

- 阅读现有实现，沿用项目结构、命名和代码风格。
- 补充局部封装、示例、测试和文档。
- 分析构建、运行、互操作和资源释放问题。
- 对 API 设计、生命周期、线程模型和验证方式提出风险提示。

人类开发者的主要职责：

- 决定 public API 形态和长期架构。
- 审查 native 资源生命周期、互操作边界和线程模型。
- 验收渲染效果、工程语义和示例可用性。
- 对 AI 生成代码进行必要重构和统一风格整理。

## 2. 标准工作流

普通任务默认使用以下流程：

```text
明确目标 -> 阅读相关代码 -> 简述方案和风险 -> 小范围修改 -> 构建/运行验证 -> 总结结果
```

当任务涉及 public API、VTK 对象生命周期、渲染管线、WPF/WinForms 承载、C# 与 native 互操作或较大范围重构时，必须先输出简短方案，等待确认后再修改。

## 3. 推荐任务输入格式

后续给 AI 分配任务时，推荐使用下面的格式：

```text
目标：为 VtkSharp 增加/修复 xxx
范围：只修改 xxx 模块、xxx 示例或 xxx 文件
不做：暂不处理 xxx
验收：运行 xxx 示例/测试，应看到 xxx 或通过 xxx
注意：关注 VTK 对象生命周期、Dispose、线程、坐标系、单位或性能
```

示例：

```text
目标：为 VtkSharp 增加 vtkPolyData 的基础封装示例
范围：只新增一个示例项目或示例文件，不调整现有 public API
不做：暂不做复杂交互和 WPF 承载
验收：示例能构造简单三角面并渲染
注意：检查 mapper、actor、renderer 的对象持有关系
```

## 4. 修改前检查清单

AI 修改代码前应先检查：

- 当前项目目录结构和相关文件。
- 是否已有相似封装、示例或命名模式。
- 是否存在项目级构建脚本、测试命令或示例启动方式。
- 当前工作区是否有未提交或用户手动修改内容。
- 本次修改是否会影响 public API、二进制兼容或示例行为。

除非明确要求，不应进行无关重构、批量格式化、重命名命名空间或移动大量文件。

## 5. VTK 封装规则

封装 VTK 类型时优先考虑以下问题：

- native 对象由谁创建，谁释放。
- C# 对象是否需要实现 `IDisposable`。
- 是否存在托管对象被 GC 后 native 侧仍持有指针或回调的问题。
- VTK 引用计数与 C# 对象生命周期是否一致。
- 是否需要保留托管委托引用，避免回调失效。
- handle、指针、字符串、数组和结构体传递是否有明确所有权。
- public API 是否暴露了过多 native 细节。

如果生命周期不确定，优先在方案中指出风险，不要用过度复杂的异常捕获掩盖问题。

## 6. C# 代码风格

默认使用现代 .NET 风格：

- 启用并尊重 nullable reference types。
- 优先使用清晰的类型、记录类型、模式匹配和 `using`/`await using`。
- 异步代码使用 `async`/`await`，避免阻塞 UI 线程。
- 资源对象应有明确释放路径。
- 不为了“新语法”牺牲可读性和稳定性。
- 不添加过度防御式代码，复杂前置条件可用断言或简短注释说明。

## 7. 桌面 UI 与承载规则

涉及 WPF、WinForms 或 DevExpress 承载时，应特别检查：

- UI 控件只能在 UI 线程访问。
- VTK 渲染窗口与宿主控件的生命周期是否一致。
- 高 DPI、窗口缩放和布局变化下是否可用。
- 绑定是否可能产生运行时错误。
- DevExpress 控件应沿用项目已有主题、命令和绑定方式。

涉及 UI 改动时，尽量提供最小可运行示例或说明手动验证步骤。

## 8. 示例策略

图形库项目的示例应承担一部分验收职责。建议示例分层维护：

- 最小渲染示例：验证窗口、renderer、actor、mapper 基础链路。
- 几何构造示例：验证点、线、面、polydata、mesh 等数据结构。
- 交互示例：验证相机、拾取、选择、高亮和事件。
- 桌面承载示例：验证 WPF/WinForms/DevExpress 集成。
- 生命周期示例：验证重复创建、销毁、释放和重新渲染。

AI 新增封装能力时，优先同时补充一个小示例。示例应尽量短、可运行、目标单一。

## 9. 测试与验证

完成修改后应尽量执行相关验证：

```powershell
dotnet build
dotnet test
```

如果项目有特定示例启动方式，应运行对应示例。对于无法自动化验证的三维渲染效果，应至少说明：

- 运行哪个示例。
- 预期看到什么几何或交互行为。
- 哪些边界条件没有验证。

几何、工程计算和渲染相关代码应重点关注：

- 空数据、单点、共线、共面、退化面。
- 极大/极小坐标。
- 容差处理。
- 坐标系和单位转换。
- 大数据量性能。

## 10. 复盘清单

每完成一个较大的迭代，可按以下问题复盘：

- 本次目标是否足够明确。
- AI 是否在修改前充分阅读了现有实现。
- 是否出现了无关改动或过度设计。
- public API 是否稳定、直观、可维护。
- native 资源生命周期是否清楚。
- 示例是否真正覆盖了新增能力。
- 构建、测试或手动验证是否充分。
- 哪些规则应该写回本文档。

## 11. 常用提示词模板

### 局部实现

```text
请先阅读相关代码，沿用当前项目结构和命名风格。
目标是实现 xxx。
修改范围限制在 xxx。
如果涉及 public API、VTK 对象生命周期或线程模型，请先说明方案和风险，等我确认后再改。
完成后运行相关构建或测试，并总结验证结果。
```

### 代码审查

```text
请从工程软件开发角度审查这次改动，优先关注：
1. VTK/native 对象生命周期
2. public API 设计
3. C# 与 native 互操作边界
4. 线程和 UI 访问
5. 示例和测试是否足够
请先列问题，再给建议，不要直接修改文件。
```

### 示例补充

```text
请为 xxx 能力补充一个最小可运行示例。
示例目标单一，代码尽量短。
不要顺手重构其他示例。
完成后说明如何运行，以及预期看到的渲染结果。
```

### 构建或运行错误排查

```text
请按系统化调试方式排查这个错误。
先复述现象和约束，再定位最小相关代码。
不要先猜测修复。
找到原因后，给出最小修改方案并说明验证方式。
```
