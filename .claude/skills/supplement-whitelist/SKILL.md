# Supplement Whitelist from Reference Exports

Follow this workflow exactly to extract VTK class names from a reference directory of `*_export_gen.cpp` files and add missing classes to the VtkSharp binding whitelist.

All commands run from repository root.

CLI binary: `dotnet run --project generator/VtkSharp.Generator.Cli --`

## Prerequisites

- A reference directory containing `*_export_gen.cpp` files (e.g. from BrdiVtkNet or another VTK .NET binding project)
- Each file `#include`s a VTK class header — the class name extracted from the `#include` line is the sole input for this workflow

## 1. Scan reference directory

List `*_export_gen.cpp` files in the reference directory. Read each one and extract the class name from the `#include` line — the text between `<` and `.h>`. The file may export any number of functions; the only information consumed by this workflow is the VTK class name in the `#include` directive:

```cpp
// <auto-generated/>
#include "pch.h"
#include <vtkClassName.h>       ← extract class name from here

VTK_NET_API vtkClassName* vtkClassName_New() { return vtkClassName::New(); }
```

→ Extract `vtkClassName` from `<vtkClassName.h>`.

Do NOT read the companion `*_export.cpp` files (no `_gen` suffix) — they only contain `#include "pch.h"` and the header, no exports.

## 2. Verify class existence and check whitelist coverage

For each extracted class name, run BOTH checks:

### 2a. Verify class exists in VTK headers

```bash
dotnet run --project generator/VtkSharp.Generator.Cli -- inspect-class <ClassName> --format json
```

Use `--format json` to get structured output containing `Name`, `Module`, `Header`, `BaseClassName`, and `HasStaticNew`.

If this fails with an error (class not found / inspection failure), the class does not exist in the current VTK installation → **skip it**.

Record for each valid class:
- `Name`
- `Module` — the authoritative VTK module (resolved from VTK class hierarchy)
- `Header` (e.g. `vtkInteractorStyleSwitchBase.h`)
- `BaseClassName` (for diagnosing build failures in step 8)

### 2b. Check if already whitelisted

```bash
grep -r "name: <ClassName>" generator/whitelist/
```

Build two lists:
- **Already covered** — skip these
- **Missing** — continue to step 3

## 3. Assign each missing class to its module whitelist

Use the `Module` field from step 2a as the authoritative source. This is resolved by the generator's VTK class hierarchy — it is always correct.

Add each class to `generator/whitelist/<Module>.yml`.

**Important**: The reference directory name is NOT the module. `inspect-class --format json` returns the true module (e.g. `vtkInteractorStyleSwitchBase` → `"Module": "vtkRenderingCore"`, not `vtkInteractionStyle`).

## 4. Add missing classes to whitelist

For each missing class, add a minimal entry to the appropriate `generator/whitelist/<ModuleName>.yml` file:

```yaml
  - name: <ClassName>
    header: <HeaderFilename>
    functions: []
```

Rules:
- Insert alphabetically within the `classes:` list (by `name` field)
- Use the exact `Header` value from `inspect-class` output (step 2a)
- Use `functions: []` — the generator handles `New()` factory, inheritance chain, and interop methods automatically
- Edit with the `Edit` tool (exact string replacement) — never rewrite entire files

**If the module whitelist file does not exist**, create it with:

```yaml
# yaml-language-server: $schema=../schemas/vtksharp.whitelist.schema.json

module: <ModuleName>
classes:
  - name: <ClassName>
    header: <HeaderFilename>
    functions: []
```

## 5. Normalize

After all manual YAML edits, run normalization to fix formatting, sorting, and consistency:

```bash
dotnet run --project generator/VtkSharp.Generator.Cli -- normalize-whitelist
```

This ensures:
- Classes are sorted alphabetically within each module
- YAML indentation is consistent
- No trailing whitespace or formatting artifacts from manual edits

Review changes with `git diff generator/whitelist/` to confirm only intended modifications.

## 6. Validate

```bash
dotnet run --project generator/VtkSharp.Generator.Cli -- validate-whitelist
```

This validates against both JSON Schema and VTK headers. It catches:
- **Module mismatch**: "Class 'vtkXxx' belongs to module 'vtkYyy' in hierarchy but is declared in 'vtkZzz' whitelist." → move the class to the indicated module file and re-run from step 5.
- **Duplicate entries**: class already exists in another module whitelist
- **Schema violations**: missing required fields, wrong types

Loop (normalize → validate → fix) until validation passes.

## 7. Regenerate bindings

```bash
dotnet run --project generator/VtkSharp.Generator.Cli -- generate-bindings --output-root src
```

This outputs:
- C# binding files → `src/bindings/VtkSharp/<Module>/<ClassName>_gen.cs`
- C++ export files → `src/native/src/<Module>/<ClassName>_export_gen.cpp`

## 8. Build and resolve inheritance gaps

```bash
dotnet build src/bindings/VtkSharp.slnx
```

If compilation fails with a missing base class:

```
error CS0246: The type or namespace name 'vtkXxxBase' could not be found
```

This means the newly added class inherits from a base class that is not yet whitelisted. To resolve:
1. Note the missing base class name from the error
2. Use `inspect-class <MissingBaseClass>` to get its header and verify it exists
3. Add it to the correct module's whitelist (return to step 4)
4. Re-run normalize → validate → regenerate → build

This can happen because a derived class and its parent may be in different VTK modules, and the reference directory only covered the derived classes.

## 9. Final verification

```bash
dotnet run --project generator/VtkSharp.Generator.Cli -- generate-bindings --check
```

Must print: `Generated output is up to date.`

## 10. Summary

Report:
- Reference directory scanned
- Total `*_export_gen.cpp` files found
- Classes skipped (not found in VTK headers)
- Classes already in whitelist (count)
- Classes newly added, with their assigned modules (after any module reassignment in step 6)
- Any base classes discovered and added during step 8
- Build result (success/failure, error/warning count)

## Rules

- **Always verify class existence** via `inspect-class` before adding to whitelist.
- **Use `inspect-class` text output** (not `--format json`) to get the `Header` field.
- **Run `normalize-whitelist`** after every batch of manual YAML edits, before validation.
- **`validate-whitelist` is the module authority** — it cross-references against the VTK class hierarchy. If it says a class belongs to a different module, move it.
- **Add `functions: []`** for class stubs from reference exports (they only export `New()`).
- **Insert alphabetically** within the `classes:` list of each whitelist file.
- **Do not duplicate** — check `grep` before adding.
- **If build fails due to missing base class**, add the base class (via the same workflow) and regenerate.
- **Whitelist files are YAML**, maintain exact indentation (2-space increments).
- Do NOT modify `generator/whitelist/` files outside the scope of the missing classes being added.
